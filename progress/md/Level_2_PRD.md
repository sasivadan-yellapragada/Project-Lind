# Level 2 PRD - Browse, Search, and Review UI

## Goal

Build a local web application for browsing the ClinicalTrials.gov dataset created in Level 1. Reviewers should be able to search and filter trials, open detail pages, and persist local watchlist, notes, and tags without modifying the imported source data.

## Implemented Architecture

- Frontend: React + Vite
- Backend: Express + Node.js REST API
- Source database: read-only SQLite `trials.db`, generated by `clinical_trials.py`
- Reviewer database: writable SQLite `user_data.db`, created by the backend
- Styling: Tailwind CSS

`trials.db` is required for meaningful L2 review. The backend can create an empty schema if the file is missing, but that is only a startup fallback; the explorer needs a populated root-level `trials.db` to show real trials.

## Core User Flows

- Browse paginated trial results.
- Search trials by keyword.
- Filter by exact phase, status, sponsor, and condition.
- Open trial details with summaries, descriptions, eligibility, conditions, and interventions.
- Add or remove a trial from the watchlist.
- Create, edit, and delete notes.
- Create, edit, and delete tags.
- Refresh or restart and retrieve the same watchlist, notes, and tags from `user_data.db`.

## API Contract

### `GET /api/trials`

Query parameters:

- `keyword`: optional SQL `LIKE` search.
- `phase`: optional exact phase match.
- `status`: optional exact status match.
- `sponsor`: optional exact sponsor match.
- `condition`: optional exact condition match.
- `page`: optional page number, defaults to `1`.
- `limit`: optional page size, defaults to `50`, capped at `100`.

Response:

```json
{
  "data": [
    {
      "nct_id": "NCT01234567",
      "title": "Example Trial Title",
      "condition": "Non-Small Cell Lung Cancer, Lung Cancer",
      "phase": "PHASE3",
      "status": "RECRUITING",
      "sponsor": "Example Sponsor",
      "start_date": "2025-01-15"
    }
  ],
  "meta": {
    "totalCount": 1234,
    "page": 1,
    "limit": 50,
    "totalPages": 25
  }
}
```

Other endpoints:

- `GET /api/filters`
- `GET /api/trials/:nctId`
- `GET /api/watchlist`
- `POST /api/watchlist`
- `DELETE /api/watchlist/:nctId`
- `GET /api/notes/:nctId`
- `POST /api/notes`
- `PUT /api/notes/:id`
- `DELETE /api/notes/:id`
- `GET /api/tags/:nctId`
- `POST /api/tags`
- `PUT /api/tags/:id`
- `DELETE /api/tags/:id`

## Decision Rationale

### React/Vite + Express vs FastAPI/Python

React/Vite was chosen because L2 is primarily an interactive browser workflow: searchable tables, detail routing, form state, and optimistic local interactions. Express keeps the thin API and frontend build in the same JavaScript ecosystem. FastAPI would be reasonable, especially because Level 1 is Python, but the L2 backend currently only reads SQLite rows and persists simple reviewer data, so adding a second web runtime was unnecessary.

### Separate `user_data.db`

Keeping reviewer annotations in `user_data.db` protects the source trial snapshot from mutable local workflow data. It also makes refreshing or regenerating `trials.db` safer because user notes, tags, and watchlist entries are not stored inside the source corpus.

The tradeoff is important: a separate SQLite database does not provide a normal database-level foreign key from `user_data.db.nct_id` to `trials.db.nct_id`. L2 enforces integrity in the application. `backend/index.js` uses `requireExistingTrial` before `POST /api/watchlist`, `POST /api/notes`, and `POST /api/tags`; invalid trial IDs return `404 {"error":"Trial not found"}` and no user row is created.

If database-enforced referential integrity becomes more important than file separation, the simpler design is one SQLite or managed relational database with source tables and separate user tables, using foreign keys from user tables to `trials(nct_id)`.

### SQL `LIKE` Search vs SQLite FTS or Indexed Search

L2 uses SQL `LIKE` because the current goal is local validation with a manageable corpus and transparent behavior. It avoids the setup and ranking decisions required by SQLite FTS.

SQLite FTS or indexed search should be revisited for Level 3 if the dataset grows, queries become slow, or users need relevance ranking, stemming, highlighting, or more advanced text matching.

### Search Fields

Keyword search includes title, sponsor, brief summary, detailed description, and condition names. Eligibility criteria is visible on the detail page but excluded from broad L2 keyword search because it can be long and noisy without ranking or highlighting.

### Exact Filters

Phase, status, sponsor, and condition filters use exact matches. The UI receives canonical values from `GET /api/filters`, so exact matching gives predictable narrowing. Alias-aware or normalized sponsor and condition matching is deferred until there is a deliberate normalization strategy.

### Offset/Page Pagination

L2 uses page-number pagination with SQL `LIMIT` and `OFFSET`. This is easy to review manually and supports next, previous, first, last, and direct page navigation. Cursor pagination would be better for very large or frequently changing datasets, but the L2 source database is local and effectively static during review.

### Scraper Artifacts

The `scraper/` files were exploratory Puppeteer artifacts, not part of L2. They have been removed so the repository reflects the implemented app rather than temporary investigation output.

## Validation Note

The demo should validate:

- The backend reads from a populated `trials.db`.
- `/api/trials` returns `data` and `meta`.
- Search and exact filters can be combined.
- Detail pages retrieve full trial data.
- Watchlist, notes, and tags persist in `user_data.db`.
- Invalid trial IDs are rejected before user-data writes.

Targeted persistence proof:

1. Add a valid trial to the watchlist from its detail page.
2. Add one note and one tag to the same trial.
3. Refresh the browser and confirm all three are still visible.
4. Restart the backend and confirm the same watchlist entry, note, and tag are still visible.
5. Confirm `/api/watchlist`, `/api/notes/<nct_id>`, and `/api/tags/<nct_id>` return the saved data.

This proves the UI is not relying only on React state; reviewer data is retrieved from `user_data.db`.
